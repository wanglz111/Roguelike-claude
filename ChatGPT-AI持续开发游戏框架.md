# AI持续开发游戏框架

## 1. 这套框架要解决什么问题

你的目标不是一次性让 AI 写一个小游戏，而是建立一个可以被 AI 长期接手、持续扩展、持续修复、持续迭代的游戏工程。

核心目标有 4 个：

1. 让游戏始终保持“可运行、可扩展、可维护”。
2. 让 AI 每一轮只做小步增量开发，而不是重写整个项目。
3. 让 AI 能根据文档、任务清单、架构规则长期接力开发。
4. 让内容层和系统层都可以无限扩展，适合 Roguelike / 魔塔 / CLI RPG。

这本质上不是“让 Claude 帮我写代码”，而是：

> 把 Claude / ChatGPT 变成这个项目的持续开发代理人。

## 2. 你真正要做的事情

你要做的是搭一个适合 AI 接力的游戏生产系统。

AI 在这个系统里的职责：

1. 读取项目规则。
2. 理解当前架构和已有功能。
3. 选择或接收下一个开发任务。
4. 实现最小可交付版本。
5. 运行测试或自检。
6. 更新文档、路线图、任务清单。
7. 为下一轮开发补充上下文。

所以这个项目不是单纯的“游戏代码仓库”，而是：

1. 一个游戏仓库。
2. 一个 AI 开发协议仓库。
3. 一个长期迭代任务系统。

## 3. 适合这套机制的游戏类型

最适合的是：

1. 文字 Roguelike
2. 魔塔 / 爬塔 RPG
3. 数据驱动的地牢探索游戏
4. 可无限追加内容的增量 RPG

推荐原因：

1. 系统可以拆模块。
2. 内容可以持续新增。
3. 地图、怪物、装备、事件都天然适合数据驱动。
4. AI 很适合做“每次增加一点内容和一套规则”。

推荐玩法循环：

```text
探索 -> 遭遇 -> 战斗 -> 获得奖励 -> 成长 -> 进入下一层 -> 解锁新系统
```

## 4. 核心设计原则

### 4.0 多语言支持

游戏支持中英文双语，所有面向玩家的文本内容都应提供中英文版本，包括：

1. 界面提示和菜单
2. 怪物名称和描述
3. 物品名称和描述
4. 事件文本
5. 战斗日志
6. 系统消息

### 4.1 小步迭代

每轮只新增一个小系统，或者对一个已有系统做一次明确增强。

好的任务例子：

1. 增加经验和升级系统
2. 新增装备栏与装备加成
3. 为怪物增加技能标签
4. 给地图加入商店事件
5. 增加存档系统

不好的任务例子：

1. 重写整个战斗系统
2. 一次性做完整剧情、地图、职业、UI、联网

### 4.2 先系统，后内容

先把框架搭稳，再往里面灌内容。

顺序建议：

1. 基础循环
2. 战斗
3. 成长
4. 掉落
5. 地图
6. 事件
7. 装备
8. 技能
9. 职业
10. 剧情和分支内容

### 4.3 数据驱动

怪物、装备、地图、事件都尽量放到 `content/` 下，而不是硬编码在逻辑文件里。

好处：

1. AI 更容易安全扩展内容。
2. 系统逻辑和内容数据可以分离。
3. 后续切换为脚本生成内容更容易。

### 4.4 文档驱动开发

AI 每轮都必须读文档。

最关键的不是提示词长度，而是这几个文件始终真实、最新：

1. `ai_dev/dev_rules.md`
2. `ai_dev/roadmap.md`
3. `ai_dev/next_tasks.md`
4. `README.md`
5. `CHANGELOG.md`

## 6. AI 持续开发闭环

每一轮开发都遵循固定闭环：

```text
读取规则 -> 读取架构 -> 读取当前任务 -> 实现最小变更
-> 运行测试 -> 修复问题 -> 更新文档 -> 生成下一批任务
```

这个闭环的重点不是“自动”，而是“稳定”。

一个能长期工作的 AI 开发流程，必须满足：

1. 每次输入上下文足够小。
2. 每次变更范围足够清楚。
3. 每次都留下下一轮可接力的信息。

## 7. AI 开发协议文件应该写什么

### 7.1 `ai_dev/dev_rules.md`

定义 AI 的长期行为边界。

应该包含：

1. 不允许重写整个项目。
2. 不允许破坏已有存档格式，除非同步更新迁移逻辑。
3. 所有新增功能必须配测试或最小验证。
4. 所有改动必须同步更新文档。
5. 优先做最小闭环，不做半成品大系统。

### 7.2 `ai_dev/roadmap.md`

定义 3 个层级：

1. 当前版本目标
2. 中期系统目标
3. 远期扩展方向

示例：

```text
Milestone 1: 可玩的单层战斗循环
Milestone 2: 多层探索 + 存档
Milestone 3: 装备、技能、商店
Milestone 4: 职业和事件分支
Milestone 5: 可持续扩展内容生产
```

### 7.3 `ai_dev/next_tasks.md`

任务要足够细，最好每条能在一次会话中完成。

示例：

```text
1. 新增 Player.exp_to_next_level 逻辑
2. 为战斗结果增加掉落奖励
3. 新增楼层生成器的最小实现
4. 为事件系统增加 Merchant 事件
5. 增加 save/load 的 JSON 持久化
```

### 7.4 `ai_dev/session_log.md`

记录每一轮 AI 做了什么。

至少写清：

1. 本轮目标
2. 改了哪些文件
3. 完成了什么
4. 遗留问题
5. 建议下一步

## 8. 什么样的代码架构适合 AI 长期维护

适合 AI 的架构不一定是最炫的架构，而是：

1. 文件职责明确
2. 命名稳定
3. 模块边界清晰
4. 状态流简单
5. 测试容易补

建议遵守：

1. 一个文件只承担一个主要职责。
2. `game/` 放系统逻辑，`content/` 放数据。
3. 复杂状态集中到 `GameState`。
4. 随机逻辑尽量可注入，方便测试。
5. 所有数值规则尽量集中配置，不要散落硬编码。

## 9. 无限扩展靠什么实现

真正的“无限开发”不是无限写代码，而是无限增加“可插入的新模块和新内容”。

可持续扩展的方向：

1. 新怪物类型
2. 新装备池
3. 新楼层规则
4. 新事件模板
5. 新职业
6. 新技能树
7. 新资源系统
8. 新 Boss 机制
9. 新剧情分支
10. 新局外成长系统

推荐按两层扩展：

1. 系统层扩展
2. 内容层扩展

系统层要慢，内容层可以快。

## 10. 版本推进建议

### Phase 1: 最小可玩版本

目标：

1. 创建角色
2. 进入楼层
3. 遇怪战斗
4. 战斗结算
5. 获得经验和金币
6. 继续下一层或死亡结束

### Phase 2: 基础成长版本

目标：

1. 升级
2. 装备
3. 掉落
4. 简单事件
5. 存档

### Phase 3: 稳定扩展版本

目标：

1. 多种怪物模板
2. 商店和神殿
3. 技能系统
4. 楼层主题
5. Boss 层

### Phase 4: 长线内容版本

目标：

1. 职业分支
2. 稀有装备词缀
3. 特殊事件链
4. 长线挑战
5. 周目或局外成长

## 11. 这套方法的关键约束

为了防止 AI 把项目越写越乱，必须加入硬约束：

1. 不允许无计划新增大系统。
2. 不允许同时重构架构和新增玩法。
3. 不允许跳过测试或验证直接堆功能。
4. 不允许文档落后于代码。
5. 不允许任务粒度过大。

## 12. 你现在最该建立的不是代码，而是秩序

如果你真的想让 Claude 或 ChatGPT 无限接力开发游戏，最重要的不是第一版代码多强，而是：

1. 规则清楚
2. 任务清楚
3. 架构清楚
4. 版本目标清楚
5. 每轮交付清楚

有了这 5 个条件，AI 才能长期稳定工作。

没有这些条件，AI 只会不断重写、漂移、失忆。

## 13. 一句话总结

这套框架的本质是：

> 用“文档 + 规则 + 任务 + 小步迭代”的方式，把 AI 变成一个可以长期接管 Roguelike 项目开发的工程代理。

下一步最合理的落地方式是：

1. 先建立 `ai_dev/` 规则体系。
2. 再搭最小可玩 CLI Roguelike 骨架。
3. 然后让 Claude 按任务清单一轮一轮开发。

配套文档见：

1. `开发文档.md`
2. `Claude-无限开发-Prompt-Framework.md`
