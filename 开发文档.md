# 开发文档

## 1. 项目定位

本项目是一个可被 AI 持续迭代开发的 CLI Roguelike / 魔塔式 RPG。

目标不是快速做一个 Demo，而是建立一个能长期扩展的工程底座，让 Claude 或 ChatGPT 能在明确规则下持续开发、修复和补内容。

## 2. 产品目标

### 2.1 核心体验

玩家在命令行中完成以下循环：

```text
进入楼层 -> 触发遭遇 -> 战斗 / 事件 -> 获得成长 -> 进入下一层
```

### 2.2 语言支持

游戏支持中英文双语，所有文本内容（包括界面、提示、怪物名称、物品描述等）都应提供中英文版本。

### 2.3 长期目标

1. 支持无限楼层扩展。
2. 支持持续新增怪物、装备、事件、技能、职业。
3. 支持存档和版本迭代。
4. 支持 AI 基于文档进行长期接力开发。

## 3. 设计边界

当前优先级：

1. CLI 玩法优先
2. 单机优先
3. 文本和数值系统优先
4. 可维护性优先于视觉表现

当前不优先：

1. 图形界面
2. 联机功能
3. 重型剧情演出
4. 复杂实时战斗

## 4. 架构原则

### 4.1 模块职责清晰

推荐目录：

```text
game/      核心规则与系统
content/   数据配置
cli/       命令行表现层
tests/     自动化测试
ai_dev/    AI 开发协议与任务系统
docs/      说明文档
```

### 4.2 数据驱动

以下内容尽量放入配置文件：

1. 怪物模板
2. 掉落表
3. 装备模板
4. 事件模板
5. 技能定义
6. 楼层主题

### 4.3 最小状态中心

建议统一由 `GameState` 持有核心状态：

1. 玩家属性
2. 当前楼层
3. 背包
4. 装备
5. 已触发事件
6. 战斗结算结果
7. 随机数种子或运行时上下文

### 4.4 可测试

以下逻辑必须尽量纯函数化或可注入依赖：

1. 伤害计算
2. 掉落计算
3. 升级计算
4. 事件选择
5. 楼层生成

## 5. 建议模块拆分

### 5.1 核心模块

`engine.py`
作用：推进主循环、处理状态切换、调度各系统。

`game_state.py`
作用：集中描述一次游戏运行的状态。

`player.py`
作用：玩家基础属性、成长接口、资源结算。

`monster.py`
作用：怪物模板实例化、行为和标签。

`combat.py`
作用：战斗回合推进、伤害计算、战斗结果。

`floor.py`
作用：楼层定义、遭遇生成、楼层主题。

`event_system.py`
作用：非战斗事件、商店、神殿、陷阱、宝箱。

`inventory.py`
作用：背包、装备栏、物品使用。

`progression.py`
作用：经验、等级、属性成长、技能解锁。

`save_load.py`
作用：存档序列化、版本迁移、读档恢复。

### 5.2 CLI 模块

`main.py`
作用：程序入口。

`renderer.py`
作用：将状态渲染成文本界面。

`input_handler.py`
作用：命令解析和交互行为分发。

## 7. 版本路线图

### 7.1 V0.1 最小可玩

交付要求：

1. 可启动游戏
2. 可创建初始角色
3. 可进入楼层并触发战斗
4. 可战斗结算
5. 可死亡或进入下一层

### 7.2 V0.2 成长闭环

交付要求：

1. 经验和升级
2. 金币和掉落
3. 背包和装备
4. 基础存档

### 7.3 V0.3 扩展闭环

交付要求：

1. 多楼层主题
2. 多种事件
3. 商店
4. Boss 层
5. 更完整测试

### 7.4 V0.4 长线扩展

交付要求：

1. 职业分支
2. 技能树
3. 稀有掉落
4. 局外成长
5. 周目机制

## 8. AI 开发工作流

每次开发会话必须遵守下面流程：

1. 阅读 `ai_dev/dev_rules.md`
2. 阅读 `ai_dev/architecture.md`
3. 阅读 `ai_dev/roadmap.md`
4. 阅读 `ai_dev/next_tasks.md`
5. 选择一个最小任务
6. 实现代码
7. 补充测试或最小验证
8. 更新文档
9. 写入 `ai_dev/session_log.md`
10. 重排 `ai_dev/next_tasks.md`

## 9. Definition of Done

一个任务只有满足以下条件才算完成：

1. 功能已实现
2. 不破坏现有主流程
3. 有最小验证方式
4. 文档已更新
5. 下一步任务已补充

## 10. 测试策略

优先覆盖：

1. 战斗结果正确性
2. 升级和属性成长
3. 掉落逻辑
4. 事件分支
5. 存档读写

建议测试层次：

1. 单元测试覆盖核心数值函数
2. 集成测试覆盖完整游戏循环
3. 回归测试覆盖关键历史问题

## 11. 文档治理要求

必须长期维护的文件：

1. `README.md`
2. `CHANGELOG.md`
3. `ai_dev/dev_rules.md`
4. `ai_dev/roadmap.md`
5. `ai_dev/next_tasks.md`
6. `ai_dev/session_log.md`

要求：

1. 文档不能落后于真实代码状态。
2. 新系统上线必须补文档。
3. 被废弃的设计必须明确标记。

## 12. 推荐的首批任务

建议按下面顺序推进：

1. 初始化目录结构和主入口
2. 建立 `GameState`
3. 实现基础 Player / Monster 数据结构
4. 实现最小战斗循环
5. 实现楼层推进
6. 实现经验和升级
7. 实现掉落和背包
8. 实现存档
9. 实现事件系统
10. 扩展内容池

## 13. 风险与防漂移策略

### 13.1 风险

1. AI 重构冲动过强
2. 文档与代码脱节
3. 任务粒度过大
4. 数值系统失控
5. 内容增加但架构没跟上

### 13.2 应对策略

1. 每轮只做一个最小闭环
2. 先读规则再编码
3. 所有任务都写清验收标准
4. 复杂改动先写设计再动代码
5. 每轮记录变更摘要和遗留问题

## 14. 当前结论

这个项目的真正核心不是“做一个 Roguelike”，而是建立一个 AI 可持续开发的 Roguelike 工程。

所以第一优先级不是拼功能数量，而是先把以下三样建稳：

1. 架构
2. 规则
3. 任务系统

然后再让 AI 长期往上叠加玩法和内容。
